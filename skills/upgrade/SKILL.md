---
name: upgrade
description: Migrate a consumer project from Purlin submodule to plugin distribution
---

**Purlin command: shared (all roles)**

Purlin agent: Migrate a consumer project from the git submodule distribution model to the Claude Code plugin model. Removes the submodule, cleans stale artifacts, declares the plugin, and migrates configuration.

---

## Usage

```
purlin:upgrade [OPTIONS]

Options:
  --dry-run    Preview changes without modifying any files
```

---

## Execution Flow

### Step 1 -- Pre-flight

1. Verify a Purlin submodule exists (check `.gitmodules` for purlin entry).
2. Verify no uncommitted changes (`git status --porcelain`). If dirty: "Commit or stash changes first." Stop.
3. Record current submodule version: `git -C <submodule> describe --tags --always`.
4. Create a pre-upgrade safety commit: `chore(purlin): pre-upgrade snapshot`.

### Step 2 -- Preserve Project Artifacts

Verify these project-owned files are NOT in the submodule (they stay untouched):
- `features/` (all feature specs)
- `.purlin/PURLIN_OVERRIDES.md`
- `.purlin/config.json` and `.purlin/config.local.json`
- `tests/` (all test results)

### Step 3 -- Remove Submodule

```bash
git submodule deinit -f <submodule_path>
git rm -f <submodule_path>
rm -rf .git/modules/<submodule_path>
```

### Step 4 -- Clean Stale Artifacts

Delete files that were generated by the submodule model:
- `pl-run.sh` (launcher)
- `pl-init.sh` (init shim, if exists)
- `.claude/commands/pl-*.md` (old skill files installed by init)
- `.claude/agents/*.md` (old agent defs installed by init)
- `.purlin/.upstream_sha` (submodule version tracker)

### Step 5 -- Declare Plugin

Update `.claude/settings.json` to declare the Purlin plugin:

```json
{
  "extraKnownMarketplaces": {
    "purlin": {
      "source": "settings",
      "plugins": [{
        "name": "purlin",
        "source": { "source": "github", "repo": "rlabarca/purlin" }
      }]
    }
  },
  "enabledPlugins": { "purlin@purlin": true }
}
```

Merge with existing settings (preserve other entries).

### Step 6 -- Migrate Configuration

Update `.purlin/config.json`:
- Remove `tools_root` key (no longer needed).
- Remove `models` array (plugin agent handles model selection).
- Remove deprecated agent entries (architect, builder, qa, pm) if present.
- Preserve `agents.purlin` section.

### Step 7 -- Update CLAUDE.md

Replace the Purlin submodule block in `CLAUDE.md` with the plugin reference:

```markdown
# Purlin

This project uses the Purlin plugin for spec-driven development.
See .purlin/PURLIN_OVERRIDES.md for project-specific rules.
```

### Step 8 -- Update .gitignore

Remove submodule-specific entries. Ensure plugin-appropriate entries exist:
```
# Purlin
.purlin/cache/
.purlin/runtime/
.purlin_session.lock
.purlin_worktree_label
```

### Step 9 -- Remove Old SessionStart/SessionEnd Hooks

Clean `.claude/settings.json` hooks that referenced submodule scripts:
- Remove hook entries with commands referencing the old submodule path.
- Plugin hooks (declared in `hooks/hooks.json`) replace them automatically.

### Step 10 -- Verification

1. Validate all JSON files modified (`.claude/settings.json`, `.purlin/config.json`).
2. Verify `features/` directory is intact with same file count.
3. Print summary:

```
Upgrade complete: submodule -> plugin

Removed:
  <submodule_path>/ (submodule)
  pl-run.sh, .claude/commands/pl-*.md, .claude/agents/*.md

Added:
  .claude/settings.json (plugin declaration)

Preserved:
  features/ (<N> specs)
  .purlin/config.json
  .purlin/PURLIN_OVERRIDES.md

Next: Run purlin:start to verify the plugin works.
```

### Step 11 -- Commit

```
chore(purlin): migrate from submodule to plugin distribution
```

---

## Dry Run Mode

When `--dry-run` is passed, print each step's planned changes without executing them. Format:

```
[DRY RUN] Step 3: Would remove submodule at purlin/
[DRY RUN] Step 4: Would delete pl-run.sh, .claude/commands/pl-*.md (12 files)
[DRY RUN] Step 5: Would add plugin declaration to .claude/settings.json
...
```
