---
name: spec-from-code
description: Reverse-engineer 3-section specs from existing code
---

Scan a codebase and generate specs in the 3-section format (`## What it does`, `## Rules`, `## Proof`). Writes to `specs/<category>/<name>.md`.

## Usage

```
purlin:spec-from-code [directory]    Scan a directory (default: src/ or lib/ or .)
purlin:spec-from-code --resume       Resume from a previous incomplete run
```

## Phase 1 — Survey

1. Ask the user which directories to scan (default: `src/`, `lib/`, `app/`; exclude: `node_modules/`, `vendor/`, `.purlin/`, `dist/`, `build/`).
2. Scan source files. Identify:
   - Module boundaries (directories, packages, namespaces)
   - Public interfaces (exported functions, classes, routes, CLI commands)
   - Cross-cutting concerns (auth, logging, error handling, config)
   - Frameworks and dependencies

## Phase 2 — Propose Taxonomy

1. Group discovered modules into feature categories.
2. For each proposed feature, identify:
   - Name (kebab-case or snake_case)
   - Category
   - Key behaviors that should be rules
3. Present to the user for review. Adjust based on feedback.

## Phase 3 — Generate Specs

For each approved feature:

1. Read the relevant source files.
2. Extract observable behaviors — what the code does, not how.
3. Write a spec in 3-section format:

```markdown
# Feature: <name>

> Scope: <source files>

## What it does

<One paragraph describing the feature.>

## Rules

- RULE-1: <Behavioral constraint extracted from code>
- RULE-2: <Another constraint>

## Proof

- PROOF-1 (RULE-1): <Observable assertion>
- PROOF-2 (RULE-2): <Observable assertion>
```

4. Write to `specs/<category>/<name>.md`.

## Phase 4 — Finalize

1. Commit all generated specs: `git commit -m "spec: generate specs from codebase (<N> features)"`
2. Call `sync_status` to show the initial coverage state.
3. Print next steps:

```
Generated N specs in M categories.

Next:
  purlin:status      — see what needs tests
  purlin:unit-test   — write proof-marked tests
  purlin:find <name> — review a specific spec
```

## Guidelines

- **Extract behavior, not implementation.** Rules should describe what the code must do, not how it does it.
- **One feature per module boundary.** Don't create specs for internal helpers — spec the public interface.
- **Conservative rule count.** Start with 2-5 rules per feature. Users can add more via `purlin:spec`.
- **Mark generated specs.** Add a note at the top: `<!-- Generated by purlin:spec-from-code. Review and refine. -->`
