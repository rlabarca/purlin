# Implementation Notes: Critic Coordination Engine

*   This policy governs buildable tooling constraints (the Critic tool itself), not process rules. It is valid under the Feature Scope Restriction mandate.
*   The `critic_gate_blocking` flag is deprecated as a no-op. The coordination engine model replaces blocking gates with advisory action items per role. The config key is retained for backward compatibility with existing `.purlin/config.json` files.
*   FORBIDDEN patterns are optional. Not all anchor nodes need to define them.
*   The CDD decoupling (Invariant 2.10) means the CDD dashboard shows role-based columns (Architect, Builder, QA) derived from `role_status` in on-disk `critic.json` files. CDD does not compute these statuses; it reads pre-computed values from the Critic's output.
*   **[CLARIFICATION]** `Action Required: Architect` BUG routing override (Section 2.4): The `parse_discovery_entries()` function already parsed the `action_required` field but the routing logic in `generate_action_items()` did not use it. Fixed by adding a conditional check on the field value before appending to `architect_items` vs `builder_items`. When `action_required` is "architect" (case-insensitive), the BUG routes to Architect with a description prefix "Fix instruction-level bug". Default routing to Builder is unchanged. (Severity: INFO)
*   **Builder Decision Audit anchor node scan (DISCOVERY resolved 2026-02-24):** `_policy_exempt_implementation_gate()` now runs `check_builder_decisions()` on anchor node `## Implementation Notes` instead of hardcoding zero counts. Tests added for DEVIATION, DISCOVERY, and AUTONOMOUS tags in anchor nodes.
*   **[CLARIFICATION]** Targeted Scope Completeness audit (Section 2.10): Implemented as a block within `generate_action_items()` rather than a separate function, following the existing pattern for scope validation. The audit reads the feature file to parse scenario titles and visual screen names, then compares against the `change_scope` from CDD status. Only runs when `lifecycle_state == 'todo'` and `change_scope.startswith('targeted:')`. Helper `_get_feature_change_scope()` added alongside `_get_feature_lifecycle_state()`. (Severity: INFO)
*   **Section 2.12 (Verification Effort Classification):** Implementation lives in `compute_verification_effort()` in `tools/critic/critic.py`, called from `generate_critic_json()` after `compute_role_status()`. The function uses data already parsed during the Critic pass (scenarios, visual spec, regression scope, role status). A new `_parse_aft_web()` helper extracts `> AFT Web:` metadata. The `verification_effort` block is written as a top-level key in `critic.json` alongside `role_status`. Full spec and scenarios are in `features/qa_verification_effort.md`.
*   **Section 2.15 (Structural Completeness Integrity) rationale (2026-03-07):** Added after a test coverage legitimacy audit found that 7 features had hand-written `{"status": "PASS"}` stubs with zero test code (79 automated scenarios with no coverage), and 7 more had `{passed: 0, total: 0}` stubs while real tests existed in a centralized suite. The prior `check_structural_completeness()` only validated file existence + `status == "PASS"` with no count, schema, consistency, or test file checks. The 4 new invariants close these gaps: (1) `total > 0` prevents zero-test PASS claims, (2) test file existence prevents stubs without backing code, (3) internal consistency catches contradictory `status: PASS` with `failed > 0`, (4) schema minimum fields invalidate the bare `{"status": "PASS"}` format. A corresponding Anti-Stub Mandate was added to `BUILDER_BASE.md` to prevent hand-written stubs at the process level.
*   **All-manual feature exemption (2026-03-08):** Added to Section 2.15 after Phase 2 implementation revealed that `release_record_version_notes` (an all-manual feature with zero automated scenarios) was incorrectly showing structural_completeness FAIL after its stub was deleted. Features with zero automated scenarios have nothing to test automatically — requiring `tests.json` for them produces false FAILs. The exemption reports PASS with `"N/A - no automated scenarios"` and allows Builder DONE without `tests.json`.
*   **Section 2.16 (QA Verification Integrity):** Implemented via three changes: (1) `_has_testing_phase_commit()` helper searches git history for `[Ready for` commits referencing the feature file, (2) `compute_role_status()` adds a `bypassed_verification` check after the TESTING-with-manual condition — when a COMPLETE feature has manual scenarios but no TESTING-phase commit, `qa_status` is set to `TODO`, (3) `compute_verification_effort()` lifecycle gating updated to treat "COMPLETE with bypassed QA" equivalently to TESTING. Internal `_bypassed_verification` flag is passed through `role_status` dict and cleaned up before JSON output. Action item generation in `generate_action_items()` independently detects bypassed verification and emits a HIGH-priority QA item with category `bypassed_qa_verification`.
*   **Section 2.12 Strong Match Threshold:** `traceability.py` now defines `STRONG_MATCH_THRESHOLD = 3` alongside `MATCH_THRESHOLD = 2`. `match_scenario_to_tests()` returns dicts `{"name": str, "overlap": int}` instead of plain strings. `run_traceability()` computes a `weak_matches` list (matches with overlap < 3). `critic.py` propagates `_weak_matches` through the implementation gate and emits MEDIUM-priority `weak_keyword_match` action items. Gherkin keywords (`when`, `given`, `then`) are excluded from the stop words to prevent false keyword overlap.
*   **Section 2.9 Figma Dev Status Advisory Gate:** Implemented via `extract_figma_status()` helper that parses `> Figma Status:` from blockquote metadata, and a check in `generate_action_items()` that emits a LOW-priority PM item with category `figma_design_not_ready` when `lifecycle_state == 'todo'` and `figma_status == 'Design'`. The status value is stored as `_figma_status` in the critic result dict (internal key, excluded from JSON output).
*   **Section 2.4 Generic BUG Action Required routing (2026-03-19):** Broadened BUG routing from Architect-only override to any-role override. Previously only `Action Required: Architect` was checked; now `Action Required: QA` routes to QA items and `Action Required: PM` routes to PM items. The Architect route preserves the "Fix instruction-level bug" description prefix for backward compatibility. Added tests for QA and PM routing paths.
*   **Section 2.14 Section Heading Migration (2026-03-22):** Updated `parse_scenarios()`, `_has_explicit_none_manual()`, and `_has_explicit_none_automated()` to accept both old (`### Automated Scenarios`, `### Manual Scenarios`) and new (`### Unit Tests`, `### QA Scenarios`) section headings. Both are parsed identically per spec requirement.
*   **Section 2.14 @auto Tag Detection (2026-03-22):** Added `is_auto` field to scenario dicts returned by `parse_scenarios()`. Detects `@auto` suffix on `#### Scenario:` headings under `### QA Scenarios`. The tag is stripped from the stored title. `compute_verification_effort()` now classifies `@auto`-tagged scenarios into the `auto` category (QA-owned but automatable) rather than manual. Added `auto` key to the `verification_effort` output block.
*   **Section 2.14 QA AUTO Status (2026-03-22):** Added `AUTO` to the QA status precedence chain in `compute_role_status()`: `FAIL > DISPUTED > TODO > AUTO > CLEAN > N/A`. A feature in TESTING state where all manual scenarios have `@auto` tag (no non-auto manual scenarios) gets `qa: AUTO`. Also added `AUTO` to the nonterminal states in the consistency check so it generates fallback action items when needed.
*   **Section 2.14 Verification Effort Summary Format (2026-03-22):** Updated summary generation in `compute_verification_effort()` to follow the spec: "N manual" when manual items exist, "N auto" when only auto items exist.
*   **Section 2.16 Abbreviated Format Support (2026-03-22):** Updated `_has_testing_phase_commit()` and `_has_verified_complete_commit()` to detect both canonical (`[Ready for \w+ features/<name>.md]`, `[Complete features/<name>.md]`) and abbreviated (`[Ready for Verification]`, `[Ready for Testing]`, `[Complete]`) status commit formats. Abbreviated commits resolve the feature from the conventional commit scope (`<type>(<scope>):` -> `features/<scope>.md`) via the new `_resolve_abbreviated_scope()` helper. Only applies when the resolved file exists on disk.
*   **Section 2.17 Regression Guidance suppression (2026-03-22):** Updated `generate_action_items()` regression guidance check to use `_has_pending_regression_guidance()` helper that checks for `tests/qa/scenarios/<name>.json` existence and `> Regression Coverage: Yes` metadata before generating action items. Previously the action item was always generated when the section existed. Also added QA CLEAN gate in `compute_role_status()` using the same helper per `critic_role_status.md` Section 2.7.
*   **[CLARIFICATION]** Internal key cleanup (2026-03-22): Added `_figma_status` to the internal-key cleanup in `generate_critic_json()`. The companion file previously stated it was "excluded from JSON output" but the `pop()` call was missing. (Severity: INFO)
