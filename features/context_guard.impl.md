# Implementation Notes: Context Guard

**[CLARIFICATION]** Session detection uses the `session_id` field from Claude Code's PostToolUse hook stdin JSON input as the primary session identifier. Falls back to `$PPID` (the Claude Code process PID) when stdin parsing fails or `session_id` is absent. (Severity: INFO)

**[CLARIFICATION]** Threshold reading uses python3 for JSON parsing (consistent with project conventions). The `try/except` wrapper handles missing files, malformed JSON, and IO errors with a fallback to the default value of 45. (Severity: INFO)

The original implementation used plain `echo` which is why agents never saw the warning despite the counter working correctly. Fixed by outputting JSON with the warning in `additionalContext`. This is a Claude Code platform behavior that should be documented in the spec's Section 2.4. (Severity: HIGH)

**[CLARIFICATION]** PostToolUse hook visibility fix: BUG — Agents never saw the context guard warning because plain stdout from PostToolUse hooks is not surfaced to the agent; fixed by switching hook output to JSON with `hookSpecificOutput.additionalContext`; default threshold also corrected from 30 to 45 turns.

BUG — `/clear` does not change Claude Code's `session_id`, so the hook never detects a new session after a context clear. The turn counter persists across `/clear` invocations within the same terminal session. Fixed by adding an explicit `echo "0" > .purlin/runtime/turn_count` step to the `/pl-resume` restore flow (Step 0 in pl-resume.md). The hook's session detection is correct for actual new sessions (new terminal) — only the `/clear` + `/pl-resume` workflow was broken.

**[CLARIFICATION]** File locking uses `mkdir`-based mutex (atomic on POSIX) since `flock` is unavailable on macOS. The lock serializes parallel PostToolUse hook invocations that fire simultaneously when Claude Code processes multiple tool calls in a single response. Without this, parallel hooks read-increment-write the same count value, causing the counter to under-count by a factor of 3-5x. A 2-second stale lock timeout prevents deadlocks from crashed processes. (Severity: INFO)

BUG — Subagent counter inflation: Task-launched subagents (Explore, Plan, etc.) each run as a separate Claude Code process with a different `session_id`. The old hook treated every session_id mismatch as a new session, resetting and re-counting. But subagent tool calls also fired the PostToolUse hook and incremented the same `turn_count` file. A single Explore agent with 27 tool calls consumed 27 of the 45-turn budget. Combined with the restore flow's ~25 calls, the counter reached 52 immediately after a `/pl-resume`. Fixed by adding subagent detection: when session_id mismatches but the `turn_count` file was modified within the last 120 seconds, the hook treats it as a subagent and exits without counting or resetting. Genuinely new sessions (stale file > 120s) still trigger a proper reset. Also added a final counter reset at the end of `/pl-resume` Step 8 so the user's actual work starts from a clean budget.

BUG — Multi-agent file collision: When concurrent agents (Architect in one terminal, Builder in another) run simultaneously, they all wrote to the same `turn_count` and `session_id` files. The second agent's session_id mismatched the first agent's stored session_id, triggering false subagent detection — the hook exited silently, producing no output and no counter tracking. Fixed by using role-specific file suffixes (`turn_count_builder`, `session_id_builder`, etc.) when `AGENT_ROLE` is set. Each agent gets its own counter and session tracking. Without `AGENT_ROLE` (direct `claude` invocation), falls back to unsuffixed files for backward compatibility.

BUG — `/pl-resume` only reset `turn_count` but not `session_id`. After `/clear`, the stored session_id could be stale from a different agent or previous session, causing false subagent detection on the new session's first hook call. Fixed by updating `/pl-resume` Step 0 and Step 8 to also delete all `session_id` and `session_id_*` files alongside the counter reset.

**[CLARIFICATION]** `/clear` does not change Claude Code's `session_id`. The `session_id` is tied to the Claude Code process, not the conversation context. After `/clear`, the hook sees the same session_id, doesn't detect a new session, and preserves the old turn count. The ONLY reliable way to reset the counter after `/clear` is to explicitly delete the `session_id*` files in `.purlin/runtime/` before the next hook invocation. This is why `/pl-resume` Step 0 (added in the spec) performs this deletion. (Severity: HIGH)

**[DISCOVERY]** Claude Code does NOT pass exported environment variables to PostToolUse hooks. `AGENT_ROLE` set via `export` in launcher scripts is invisible to the hook's shell process. Fixed by persisting the role to `.purlin/runtime/agent_role` in each launcher script, and reading it as a fallback in the hook when the env var is empty. This file-based approach also survives `/clear` + `/pl-resume` since the file persists on disk. Without this fix, per-agent threshold resolution always fell back to the global default (45). (Severity: HIGH)

BUG — Output format showed `REMAINING/THRESHOLD` (counting down) instead of `COUNT / THRESHOLD used` (counting up). The companion file recorded the format change intent but the code was never updated — the hook still computed `REMAINING=$((THRESHOLD - COUNT))` and output that. Agents consistently misread `44/45` as "44 used of 45" (nearly exhausted) when it actually meant "44 remaining." Fixed by removing the REMAINING calculation and outputting `${COUNT} / ${THRESHOLD} used` directly. The subagent code path was also updated to use the same format.

BUG — Subagent detection (`IS_SUBAGENT=true`) caused `exit 0` with no JSON output, violating spec Section 2.4 "no silent exits." Fixed by reading the current `turn_count` (without incrementing) and outputting the context guard status before exiting. All code paths now produce output when the guard is enabled.

BUG — False subagent detection on new sessions: The 120-second file-age heuristic for subagent detection was unreliable. When a new session started within 120 seconds of a previous session's last tool call, the hook falsely identified it as a subagent, displaying the stale counter (e.g., "55 / 60 used" on the first turn of a fresh session). Additionally, role-based file suffixes (`turn_count_builder`) couldn't handle multiple agents of the same role running concurrently. Fixed by replacing the entire session/subagent detection system with PPID-based agent identity: each Claude Code process gets its own counter file (`turn_count_<PPID>`), and subagents are detected deterministically by session_id mismatch within the same PPID (via `session_meta_<PPID>`). `CONTEXT_GUARD_AGENT_ID` env var overrides PPID for testing. Stale files from dead processes are cleaned up automatically via `kill -0` checks. Legacy role-suffixed and unsuffixed files are removed on first run.

**[DISCOVERY]** The `/pl-resume` Step 0 and Step 8 reset instructions reference legacy file patterns (`session_id_*`, `turn_count_*` with role suffixes). With the PPID-based design, the reset should target `turn_count_$PPID` and `session_meta_$PPID`. The existing wildcard reset (`for f in turn_count_*; echo "0" > "$f"`) still works but could reset other running agents' counters. The Architect should update `/pl-resume` Step 0/Step 8 to use PPID-scoped resets. (Severity: HIGH)

**[DISCOVERY]** Spec Section 2.3 and the "Subagent detection" scenario (Section 3) reference the 120-second file-age heuristic which has been replaced. The Architect should update Section 2.3 to describe PPID-based agent identity and update the subagent scenario to reference session_meta instead of file modification timestamps. (Severity: HIGH)
